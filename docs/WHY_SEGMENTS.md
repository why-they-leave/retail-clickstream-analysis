# 이 프로젝트는 왜 세그먼트(페르소나)를 다는가

이 문서는 세그먼트 관련 개별 이슈(#1, #14~#18, #23, #26)를 하나하나 보지 않아도, "애초에 왜 이 작업을 하는지"를 한 번에 이해할 수 있도록 배경을 정리한다. 구현 방법 자체는 `docs/SEGMENT_ASSIGNMENT_DESIGN.md`와 각 리포트를 참고한다.

## 프로젝트 전체 목표와의 연결

이 프로젝트가 속한 상위 시스템은 "**페르소나-로그 결합형 퍼널 진단 기반 추천 시스템**"이다. 핵심 질문은 단순하다.

> 유저의 행동 로그(클릭, 장바구니, 구매)만 보고 추천하는 것과, 거기에 "이 유저가 어떤 소비 성향 그룹인지"라는 정보를 더해서 추천하는 것 — 후자가 실제로 더 나은가?

세그먼트는 이 질문에 답하기 위한 재료다. `rec-system` 레포의 ALS(#7)가 로그 기반 베이스라인이라면, 지금 준비 중인 LightGCN_tri(#28)는 여기에 세그먼트를 세 번째 그래프 노드로 얹어서 "페르소나 정보를 더하면 추천이 좋아지는가"를 직접 검증하는 실험이다.

## v1: LLM이 페르소나를 직접 창작 (실패 원인 확인)

최초 접근(#1, GPLR 알고리즘)은 고객별 구매 상품명과 행동 정보를 LLM에 통째로 넣고, LLM이 페르소나 taxonomy 자체를 만들도록 했다.

문제는 이 데이터셋의 상품명이 `SSD MediumBlue 149`, `Lipstick LightBlue`, `Paperback Olive`처럼 **synthetic 데이터 생성기가 만든 더미 텍스트**라는 점이다. LLM은 이 상품명에서 실제로는 존재하지 않는 라이프스타일/취향을 그럴듯하게 추론해버렸고(#14), 그 결과 페르소나가 중복되거나 synthetic 텍스트의 우연한 패턴을 반영하는 문제가 있었다.

## v2: 먼저 클러스터링, LLM은 이름만 붙임 (지금 쓰는 방식)

그래서 순서를 바꿨다.

1. **#15~#16**: 상품명이나 텍스트는 전혀 쓰지 않고, 순수 숫자 행동 피처(페이지뷰 수, 장바구니 담기율, 구매 횟수, recency, 카테고리 다양성 등)만으로 KMeans 클러스터링을 해서 `segment_id`(0~5, 6개)를 확정한다. **이 단계에는 LLM이 전혀 관여하지 않는다** — 재현 가능하고 검증 가능하다.
2. **#17**: LLM은 각 `segment_id`의 집계 통계(segment_summary)만 보고, 사람이 읽을 이름/설명을 붙이는 **라벨러 역할**로만 쓴다. "이 세그먼트가 몇 살대인지, 어떤 직업군인지"처럼 데이터에 없는 걸 추론하는 건 프롬프트 규칙으로 명시적으로 금지했다. 결과에는 `evidence`(어떤 수치를 근거로 이 이름을 붙였는지)와 `cautions`(이 이름을 과신하면 안 되는 이유)를 항상 같이 남긴다.
3. **#18**: v1과 v2 결과를 비교해서, v2가 synthetic 상품명 아티팩트 없이 더 신뢰할 수 있는 라벨을 만든다는 걸 확인했다.
4. **#23**: Full 데이터와 US-only 데이터를 따로 만들 필요가 없다는 게 확인돼서(#4), US 트랙을 제거하고 Full 데이터 단일 트랙으로 통일했다.
5. **#26**: 그동안 `segment_id`(고객별)와 `segment_name`/`description`(세그먼트별)이 서로 다른 파일에 따로 있어서 실제로 못 쓰는 상태였다. 이 둘을 고객 단위로 합쳐서 `customer_segments_labeled_all_customers.csv` 한 장으로 만들었다 — 이제야 "이 고객이 어떤 세그먼트고 왜 그런지"를 바로 조회할 수 있다.

## 중요한 전제: 세그먼트는 정답 라벨이 아니다

세그먼트/페르소나는 **행동 로그의 소비 패턴을 해석하기 위한 보조적 설명 레이어**다. 실제 유저를 대표하는 정답 라벨이 아니다. KMeans는 실루엣 점수가 낮아 "뚜렷하게 분리된 자연 집단"이라기보다 "행동 기반 근사 그룹"에 가깝고, LLM이 붙인 이름도 그 근사치를 사람이 읽기 쉽게 요약한 것일 뿐이다. 그래서 세그먼트를 쓰는 모든 후속 작업(LightGCN 포함)에서 "세그먼트가 곧 유저의 정체성"이라고 과신하면 안 된다.

또한 세그먼트 정의는 신규 데이터가 들어오면 (KMeans를 다시 학습하는 경우) 바뀔 수 있다는 한계가 있다 — 이 부분은 아직 운영 파이프라인 단계에서 다루지 않았고, 실제 재학습 주기가 정해지는 시점에 버전 관리를 검토하기로 했다(`docs/SEGMENT_ASSIGNMENT_DESIGN.md` 참고).

## 지금 이걸 쓰는 곳

- `rec-system` (별도 레포): 추천 데모에 세그먼트/페르소나 정보를 노출하는 것을 향후 계획 중 (아직 실제 연동 코드는 없음)
- `LightGCN_tri` (#28, 진행 중): 세그먼트를 유저-상품 그래프의 세 번째 노드로 추가해서, ALS 대비 "페르소나 결합"의 실제 효과를 측정
